# 要件定義書 (Requirements Document)

## はじめに
本アプリケーションは、案件ごとのスケジュール調整を効率化するための管理ツールです。
複数の候補日を提示・管理する機能、および特定の日への予定集中（オーバーロード）を可視化する機能を提供することで、少人数チームにおけるリソース配分の最適化と、調整コストの削減を目的としています。

## 製品ビジョンとの整合性
- **柔軟な調整**: 1つの予定に対して最大5つの候補日を保持できる仕組みを提供し、未確定段階の予定管理をサポートします。
- **リソース過多の防止**: 案件を横断して「1日に3つ以上の確定予定」がある日を自動抽出することで、無理なスケジューリングを未然に防ぎます。
- **モダンな開発体験**: Nuxt 4、TypeScript、Firebase、Playwright を活用し、保守性が高く、最新のWeb標準に準拠した軽量なシステムを実現します。

## 要件

### 要件 1：候補日管理とステータス遷移

**ユーザーストーリー:** 利用者として、1つの予定に対して複数の候補日を設定し、その中から1つを確定させたい。そうすれば、関係者との調整がスムーズになり、確定後の不要なデータ削除も自動化できるからだ。

#### 受け入れ条件 (Acceptance Criteria)

1. 予定登録時、利用者は最大5件まで候補日を指定できなければならない（SHALL）。
2. 候補日のうち1つを「確定」ステータスに変更したとき、システムは即座にその予定の他の候補日を削除しなければならない（SHALL）。
3. 候補日が5件を超えて登録しようとした場合、システムはエラーを表示し、登録をブロックしなければならない（SHALL）。

### 要件 2：過密スケジュールの可視化

**ユーザーストーリー:** 利用者として、全ての案件を通じて予定が立て込んでいる日を一覧で確認したい。そうすれば、リソースの限界を把握し、新たな案件の受け入れ判断が容易になるからだ。

#### 受け入れ条件 (Acceptance Criteria)

1. システムは、案件を横断して「確定（confirmed）」状態の予定が同一日に3つ以上存在する日を特定しなければならない（SHALL）。
2. 特定された過密日は、通常の予定登録ページとは別の専用ページ（過密日確認ページ）で一覧表示されなければならない（SHALL）。
3. 予定が削除またはステータス変更され、同日の確定予定が2件以下になった場合、その日は即座に過密日一覧から除外されなければならない（SHALL）。

### 要件 3：マルチデバイス対応と認証

**ユーザーストーリー:** 利用者として、PCでもスマホでも安全に自分のスケジュールにアクセスしたい。そうすれば、場所を選ばずに最新の状況を確認・更新できるからだ。

#### 受け入れ条件 (Acceptance Criteria)

1. 利用者は、Firebase Authentication を介してログインしなければシステムを利用できてはならない（SHALL）。
2. システムはレスポンシブデザインを採用し、モバイル端末（スマホ）とデスクトップ（PC）の両方で、レイアウトが崩れることなく全ての機能が操作可能でなければならない（SHALL）。

## 非機能要件

### コードアーキテクチャとモジュール性
- **単一責任の原則**: 各コンポーザブル（Composables）は、Firebase操作、日付計算、認証管理など、単一の明確な目的を持つように設計する。
- **モジュール設計**: UIコンポーネント、ビジネスロジック（Composables）、ユーティリティを明確に分離し、再利用性を高める。
- **依存関係管理**: Nuxt 4 の Auto-imports を活用しつつ、モジュール間の密結合を避ける。
- **明確なインターフェース**: TypeScript を用い、Project や Schedule の型定義を厳格に行う。

### パフォーマンス
- Vercel のエッジネットワークを活用し、初回レンダリング（LCP）を2.5秒以内に抑える。
- Firestore のクエリを最適化し、10人程度の同時アクセスでも遅延を感じさせないレスポンスを実現する。

### セキュリティ
- Firebase Auth によるユーザー識別。
- Firestore Security Rules を定義し、認証済みユーザーのみが自身の所属するデータにアクセスできるように制限する。
- 環境変数は Vercel のダッシュボード上で安全に管理し、クライアント側に不要な秘密情報を露出させない。

### 信頼性
- Firebase の無料枠（Sparkプラン）内で運用を完結させるため、過度な書き込みが発生しないよう、クライアントサイドでのバリデーションおよびバッチ書き込みを徹底する。
- Firestoreのオフラインキャッシュを活用し、一時的なネットワーク断が発生した場合でも、UI上で最後に取得したデータを表示できるようにする。
