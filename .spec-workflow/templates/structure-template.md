## コード組織の原則

1. **単一責任の原則 (Single Responsibility)**: 各ファイルは1つの明確な目的を持つべきです
2. **Mモジュール化 (Modularity)**: コードは再利用可能なモジュールとして整理されるべきです。
3. **テストの容易性 (Testability)**: テストが容易に行えるようにコードを構造化します。
4. **一貫性 (Consistency)**: コードベース内で確立されたパターンに従います。

## ダッシュボード/モニタリング構造（該当する場合）
[ダッシュボードやモニタリングコンポーネントをどのように整理するかを定義してください]

### 構成例:
```
src/
└── dashboard/          # 自己完結型のダッシュボード・サブシステム
    ├── server/        # バックエンド・サーバーコンポーネント
    ├── client/        # フロントエンド・アセット
    ├── shared/        # 共有の型定義/ユーティリティ
    └── public/        # 静的アセット
```

### 関心の分離
- ダッシュボードをコアのビジネスロジックから分離する
- 独立して動作させるための独自のCLIエントリポイントを持つ
- メインアプリケーションへの依存を最小限に抑える
- コア機能に影響を与えることなく無効化できる

## ドキュメント標準
- すべての公開APIにはドキュメントを作成すること
- 複雑なロジックにはインラインコメントを含めること
- 主要なモジュールにはREADMEファイルを設置すること
- 言語固有のドキュメント慣習に従うこと
